# handsomeW-agent 产品功能优化与演进方案

> **版本**: V1.0 | **编制日期**: 2026年4月17日 | **目标受众**: 产品负责人 / 技术团队 / 架构师

---

## 一、执行摘要

handsomeW-agent 是一款轻量级 Agent Web 应用，采用双后端架构（Spring Boot 系统服务 + Python AI 服务）与 React 前端，涵盖多 Agent 运行时、数字孪生、GraphRAG、Skill 商店等能力。当前项目处于 Beta 阶段（v0.1.0），核心能力已基本可用，但各模块之间的联动与深度集成仍有较大提升空间。

本方案从产品体验、技术架构、工程质量三个维度提出 12 项优化建议，并给出分阶段实施路线图。核心主题是：将数字孪生 + Agent + GraphRAG 三大能力真正打通，形成"空间感知 × 智能对话 × 知识图谱"的差异化产品体验。

---

## 二、现状评估

### 2.1 架构概览

当前项目采用双后端微服务架构，Python AI 服务（FastAPI）承载 Agent 运行时、工具调用、GraphRAG 等智能能力；Spring Boot 系统服务承载认证、用户、数字孪生等系统业务。前端基于 React + Vite + TypeScript + Tailwind，包含 12 个视图页面和 9 个核心组件。

### 2.2 能力成熟度矩阵

| 功能模块 | 成熟度 | 优势 | 短板 |
|---------|--------|------|------|
| 多 Agent 运行时 | ★★★★ | 支持 ReAct/Simple/Loop 三种模式，异步流式响应 | 缺少多 Agent 协作编排 |
| LLM 接入层 | ★★★★★ | 支持 10+ 服务商，基于 litellm 统一抽象 | 缺少用量统计与成本监控 |
| 数字孪生 | ★★★ | 3D 场景渲染、资产/场景/资源 CRUD | 与 Agent 对话未联动 |
| GraphRAG | ★★☆ | Neo4j + Milvus 混合检索架构已就绪 | Agent 集成未启用，数据模型待迁移 |
| Skill 商店 | ★★☆ | GitHub topic 发现 + 前端浏览 | 缺少安装闭环与社区反馈 |
| 告警流 | ★★★ | DAG 编辑器 + Neo4j 执行引擎 | 缺调试与监控能力 |
| 认证与权限 | ★★ | 基础 Token 认证 + admin/user 角色 | 缺细粒度权限和多租户 |
| System Service 迁移 | ★☆☆ | 骨架已搭建，代理模式运行 | 仅完成约 30% |
| 测试覆盖 | ☆☆☆ | — | 未发现任何测试代码 |

## 三、优化方案详述

### 3.1 Agent 对话体验升级

#### 功能 1：Agent 思维链路可视化

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 4–6 周 |

**问题**: ReactAgent 在多轮迭代中调用工具时，前端无法展示每一步的推理过程，用户难以理解 Agent 在做什么，也不方便调试。

**方案**:

- 在 ChatView 中增加"Thinking"展开面板，实时流式显示 Agent 正在执行的步骤
- 每个工具调用显示工具名称、输入参数、输出结果、耗时
- 支持折叠/展开单步详情，避免信息过载
- 后端在 AgentChatResponse 中增加 `thinking_steps` 字段，携带每步的 `tool_name`、`input`、`output`、`duration_ms`

#### 功能 2：对话上下文管理面板

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 3–4 周 |

**问题**: 用户无法主动管理对话上下文，导致长对话中信息溢出或上下文窗口浪费。

**方案**:

- 在 ChatView 侧边栏显示当前会话的 Token 使用量 / 窗口余量进度条
- 支持用户 Pin 重要消息，保证关键信息不被截断
- 支持上传文档作为临时知识库注入当前对话
- 后端 `chat_memories` 表增加 `pinned` 字段，优化检索策略优先返回 pinned 消息

#### 功能 3：Agent 模板与分享机制

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 3–4 周 |

**问题**: 用户配置好用的 Agent（模型 + 提示词 + 工具集 + 技能）后无法保存和分享。

**方案**:

- 新增 `agent_templates` 表，存储模板配置（agent_type、system_prompt、tools、skills、llm_config）
- 前端增加"保存为模板"入口，支持命名、描述、标签
- 模板市场页面，支持按场景/标签筛选、一键克隆使用
- 团队内分享链接，降低新用户上手门槛

### 3.2 核心能力深化

#### 功能 4：多 Agent 协作与编排

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 6–8 周 |

**问题**: 当前 SimpleAgent、ReactAgent、LoopAgent 独立运行，缺少多 Agent 协作能力，无法处理复杂的多步骤任务。

**方案**:

- 设计 OrchestratorAgent 编排层，支持任务分解 → 并行派发 → 结果汇聚的工作流
- 参考 CrewAI 的角色分配模式：规划 Agent、执行 Agent、审核 Agent
- 前端提供可视化 Agent 工作流编辑器（复用 AlarmFlowEditor 的 DAG 组件）
- 支持 Agent 间的消息传递、共享上下文、优先级调度

#### 功能 5：GraphRAG 深度集成

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 4–6 周 |

**问题**: 根据代码分析，`_build_prompt_with_rag()` 在 `agent_service.py` 中尚未启用，GraphRAG 数据模型仍针对食谱领域，需迁移至工业设备领域。

**方案**:

- 启用 `agent_service.py` 中的 RAG 集成代码，将 GraphRAG 作为 Agent 的一等公民工具
- 按 `docs/graph-rag-digital-twin-integration.md` 设计迁移数据模型：Equipment、Sensor、FaultMode、MaintenanceRecord
- 对话中支持知识图谱子图可视化，展示检索到的实体关系
- 实现混合检索策略：向量相似度 + 图遍历 + 关键词匹配的加权融合

#### 功能 6：Skill 商店闭环体验

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 3–4 周 |

**问题**: 当前从 GitHub topic 发现技能，但缺少安装 → 使用 → 反馈的完整闭环。

**方案**:

- 增加技能安装状态管理（installed/available/updating），前端实时反馈安装进度
- 对话中智能推荐：根据用户意图自动建议相关技能
- 用户评分与评论系统，形成社区反馈闭环
- 技能版本管理与自动更新机制

### 3.3 数字孪生与 Agent 联动

#### 功能 7：空间智能交互

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 6–8 周 |

**问题**: 数字孪生（3D 场景）与 Agent 对话目前是相互独立的模块。将其打通是本项目最核心的差异化竞争力。

**方案**:

- 在 Scene3DView 中点击设备模型时，自动唤起 Agent 对话面板，预填设备上下文
- Agent 可查询该设备的历史告警、维护记录、知识图谱关联信息
- Agent 输出结果可反向标注到 3D 场景（如高亮异常设备、显示告警热力图）
- 支持自然语言控制场景："显示所有温度异常的传感器" → 3D 视角自动定位

#### 功能 8：告警流调试与监控

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 3–4 周 |

**问题**: AlarmFlowEditor 已支持 DAG 可视化编辑，但缺少执行期的调试与监控能力。

**方案**:

- 流程执行时实时日志流，节点状态彩色标注（运行中/成功/失败）
- 节点级别断点调试：暂停在某节点查看上下文数据
- 执行历史回放：选择历史执行记录重放整个流程执行过程
- 失败节点一键重试，无需重跑整个流程

### 3.4 平台能力建设

#### 功能 9：可观测性仪表盘

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 4–6 周 |

**问题**: DashboardView 当前较简单，缺少对平台运行状态的深度洞察。

**方案**:

- Token 用量统计：按用户/Agent/模型维度展示调用量与成本趋势
- API 调用延迟分布：P50/P95/P99 延迟曲线，快速定位性能瓶颈
- Agent 任务成功率与平均迭代次数统计
- 工具调用频率排行榜，辅助优化工具集配置

#### 功能 10：细粒度权限与多租户

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 4–6 周 |

**问题**: 当前仅有 admin/user 两种角色，无法满足企业级场景需求。

**方案**:

- 设计 RBAC 模型：角色（admin/editor/viewer）+ 资源维度（Agent/场景/知识库）
- 支持团队/项目级别的资源隔离
- 审计日志：记录所有敏感操作（模型配置变更、用户权限变更等）
- 优先在 system-service 中实现，与 Java 迁移路线对齐

### 3.5 工程质量提升

#### 功能 11：测试体系建设

| 属性 | 内容 |
|------|------|
| 优先级 | 高 |
| 周期 | 持续进行 |

**问题**: 整个项目当前没有任何测试代码，在单体向微服务迁移的过程中风险较高。

**方案**:

- AI 服务：使用 pytest 为 Agent 运行时、工具调用链路、认证流程编写单元测试
- System Service：使用 JUnit 5 + Spring Boot Test 覆盖核心 API 接口
- 前端：使用 Vitest + React Testing Library 覆盖关键组件交互
- 集成测试：使用 Docker Compose 构建端到端测试环境，覆盖 Agent 对话全链路

#### 功能 12：System Service 迁移策略优化

| 属性 | 内容 |
|------|------|
| 优先级 | 中 |
| 周期 | 8–12 周 |

**问题**: 当前迁移仅完成约 30%，需要明确优先级和迁移策略以确保平稳过渡。

**方案**:

- 第一优先级：认证与用户管理完整迁至 Java（所有功能的基础）
- 第二优先级：数字孪生 CRUD + MinIO 文件服务迁至 Java
- API 版本管理：引入 `/api/v1`、`/api/v2` 前缀，避免前端频繁适配
- 大功能切换使用 Feature Flag，支持灰度发布和快速回滚

| M2（第 7-12 周） | 能力深化，打通联动 | GraphRAG 深度集成、空间智能交互 MVP、Skill 商店闭环 | Agent 可查询知识图谱、3D 场景可唤起 Agent |

## 六、关键技术实现要点

### 6.1 Agent 思维链可视化实现路径

后端改造：在 ReactAgent 的每次 tool 调用前后增加事件发射，通过 SSE（Server-Sent Events）流式推送给前端。建议在 AgentChatResponse 中新增 `thinking_steps` 字段，每个 step 包含 `step_type`（thinking/tool_call/tool_result）、`tool_name`、`input`、`output`、`duration_ms`。前端使用 EventSource 监听，通过折叠面板渲染每个步骤。

### 6.2 GraphRAG 集成路径

核心改动点在 `agent_service.py` 的 `_build_prompt_with_rag()` 方法。需要：① 启用该方法的调用；② 按 `docs/graph-rag-digital-twin-integration.md` 重构数据模型，将实体类型从食谱领域迁移到工业设备领域（Equipment、Sensor、ProductionLine、FaultMode、MaintenanceRecord）；③ 在 RuntimeQueryRouter 中增加设备专用的查询分类策略。

### 6.3 空间智能交互架构

核心思路是在 Scene3DView 和 ChatView 之间建立双向通信通道。点击 3D 设备时，通过 Zustand appStore 将 `asset_id` 和 `asset_context` 写入全局状态，ChatView 监听变化后自动注入设备上下文到 Agent 提示词。反向链路：Agent 输出包含设备 ID 时，前端解析后通知 Scene3DView 高亮对应模型。建议定义统一的 SceneAgentEvent 协议格式。

### 6.4 测试策略建议

AI 服务（Python）：使用 pytest + pytest-asyncio 测试异步 Agent 流程，httpx.AsyncClient 测试 FastAPI 接口，使用 `unittest.mock` 模拟 LLM 调用避免真实 API 费用。System Service（Java）：使用 JUnit 5 + Mockito + Spring Boot Test，重点覆盖认证流程和代理转发逻辑。前端：Vitest + React Testing Library，优先覆盖 ChatView 和 AlarmFlowEditor 的交互逻辑。

## 七、总结与建议

handsomeW-agent 的核心架构已经相当扎实，技术选型合理，具备良好的演进基础。本方案的核心主张可以概括为以下三点：

1. **"打通"是最大机会**：数字孪生 + Agent + GraphRAG 三大能力的联动是本项目的核心差异化竞争力。空间智能交互（功能 7）是最具战略价值的功能点。

2. **"可见"是信任基础**：Agent 思维链可视化（功能 1）和可观测性仪表盘（功能 9）让用户和运维团队都能"看得见"系统在做什么，是产品可用性的基础。

3. **"稳固"是持续演进的前提**：测试体系（功能 11）和 System Service 迁移（功能 12）虽然不直接面向用户，但是所有功能快速迭代的安全网。

建议立即启动 M1 阶段工作，优先落地测试体系、Agent 思维链可视化和可观测性仪表盘，为后续能力深化奠定坚实基础。
